# ExamPlanner — дизайн MVP

- **Дата:** 2026-07-08
- **Статус:** одобрен владельцем (брейншторминг), готов к плану реализации
- **Фаза:** MVP (v1)

## 1. Задача и польза

Мобильное приложение для студентов, помогающее готовиться к экзаменам: вести список тем,
отслеживать прогресс изучения и планировать время так, чтобы успеть к датам экзаменов.

Конечная цель — многопользовательский продукт. Строим слоями; **эта спека покрывает только MVP**:
одиночное локальное Android-приложение. Серверная часть и всё остальное — в разделе
[«Дальнейшие фазы»](#8-дальнейшие-фазы-роадмап-отложенного).

## 2. Область MVP (что делаем)

- Несколько экзаменов, у каждого — дата и свой список тем.
- Прогресс по теме: три статуса **Не начата → В процессе → Пройдена**; прогресс экзамена вычисляется.
- Оптимизация времени: рекомендованное время в день + дневной план «Сегодня» с учётом
  предмета (время на тему), числа тем и даты экзамена; подстройка под доступность пользователя.
- Платформа: **Android** (.NET MAUI). Данные локально в **SQLite**. Один пользователь, без сети.

## 3. Технический подход

**Выбрано: MAUI + MVVM + SQLite (sqlite-net).** Слои: `Models / Data / Services / ViewModels / Views`.
`CommunityToolkit.Mvvm` для сокращения рутины. Логика расчётов изолирована в сервисе и
покрывается юнит-тестами.

Рассмотренные альтернативы:
- *MAUI без MVVM (code-behind)* — быстрее на старте, но плохо тестируется и растёт в кашу. Отклонено.
- *MAUI + EF Core* — полноценный ORM, тяжелее нужного для локального MVP. Отклонено в пользу sqlite-net.

## 4. Модель данных (SQLite)

```
Exam    { Id, Name, Date, MinutesPerTopic, CreatedAt }
Topic   { Id, ExamId -> Exam, Title, Status (NotStarted | InProgress | Done), Position }
Settings{ AvailableHoursPerDay }        // одна строка, глобальная настройка
```

- `MinutesPerTopic` — среднее время на тему у этого экзамена (отражает «сложность предмета»).
- Прогресс экзамена **не хранится**, а вычисляется из статусов тем (нет рассинхрона).
- `Position` — порядок тем внутри экзамена (для ручной сортировки).

## 5. Экраны (Android)

1. **Главная (Dashboard)** — список экзаменов: название, дата, «12/20 (60%)», полоска прогресса,
   подсветка срочных. Сверху сводка: «Рекомендуем сегодня ≈ X ч · у тебя Y ч» + предупреждение,
   если не успеваешь.
2. **Экзамен** — темы со статусами (тап меняет статус по кругу), прогресс, темп
   «осталось N тем, D дней → ~X тем/день». Добавить/править/удалить тему.
3. **Сегодня** — сгенерированный дневной план: темы на сегодня по срочности под доступность;
   отметка «Пройдена» пересчитывает план.
4. **Настройки** — доступность (часов в день).
5. **Модалки** — добавить/править экзамен (имя, дата, мин/тема); добавить/править тему.

## 6. Логика оптимизации времени (детерминированная)

Ядро — отдельный сервис (напр. `PlannerService`), без UI, полностью тестируемый.

Для каждого экзамена `e`:
```
remaining_e   = число тем со статусом != Done
daysLeft_e    = max(1, ExamDate_e - сегодня)
needPerDay_e  = remaining_e * MinutesPerTopic_e / daysLeft_e     // минут/день
```
- **Рекомендация на день** = Σ needPerDay_e по всем экзаменам.
- **Доступность** = AvailableHoursPerDay * 60 (минут). Если рекомендация > доступности —
  предупреждение + подсветка самых горящих экзаменов (макс. needPerDay_e).
- **План «Сегодня»**: экзамены по возрастанию `ExamDate` (срочность); жадно набираем темы
  (каждая ≈ MinutesPerTopic_e), пока не заполним доступные минуты. Что не влезло — помечаем
  «не успеваешь».

### Краевые случаи (обязательны в тестах)
- У экзамена 0 непройденных тем → не участвует в рекомендации/плане.
- Дата экзамена сегодня или в прошлом → `daysLeft = 1` (весь остаток «на сегодня»), экзамен
  помечается как критический.
- Все темы Done → прогресс 100%, экзамен уходит из активных расчётов.
- Доступность = 0 → план пуст, показывается только рекомендация и предупреждение.
- Нет экзаменов/тем → пустые состояния экранов.

## 7. Тестирование

- **Юнит-тесты** на `PlannerService`: темп, рекомендация, дневной план, все краевые случаи из §6.
- Репозиторий и ViewModels тестируемы (логика вне UI).
- Ручная проверка на Android-эмуляторе (`superpowers:verification-before-completion` перед «готово»).

## 8. Дальнейшие фазы (роадмап отложенного)

> Явный список того, что **сознательно НЕ входит в MVP**, но планируется добавить позже.
> Владелец просил напомнить об этом — зафиксировано здесь под версионным контролем.

**Фаза v2 — многопользовательский бэкенд**
- ASP.NET Core Web API + база данных на сервере.
- Аккаунты и аутентификация (регистрация/вход).
- Синхронизация данных между устройствами; хостинг в облаке.

**Фаза v3 — расширение клиента и UX**
- iOS (нужен Mac для сборки/публикации).
- Пуш-уведомления и напоминания о занятиях.
- Аналитика прогресса (графики, статистика).

**Отложенные фичи планирования и оценки**
- Оценка времени на тему точнее: уровни сложности (Лёгкий/Средний/Тяжёлый) или минуты на каждую тему.
- Доступность по дням недели (в будни меньше, в выходные больше).
- Таймер учёбы (помодоро) + учёт фактически потраченного времени и статистика.
- Мини-тесты/квизы по темам и «двойная оценка» (объективная от приложения + самооценка);
  возможная ИИ-генерация вопросов (требует интернета/ИИ-API/сервера).

Каждая фаза оформляется **своей** спекой → планом → реализацией.

## 9. Допущения MVP

1. Все темы экзамена равны по объёму — время на тему одно на весь экзамен (`MinutesPerTopic`).
2. Доступность — одно число часов в день (без разбивки по дням недели).
3. Расчёты детерминированные, без ИИ и без сети.
