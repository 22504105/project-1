# Оценка времени по сложности темы — дизайн

- **Проект:** ExamPlanner
- **Фаза:** пост-MVP, инкремент планирования (Core)
- **Дата:** 2026-07-09
- **Автор:** 22504105
- **Трек:** backend/Core (параллельно frontend-агенту, который делает темизацию). UI **не** входит в этот инкремент.

## 1. Цель

Сейчас планировщик считает время грубо: все темы одинаковые — `оставшиеся_темы × exam.MinutesPerTopic`. Ввести **уровень сложности темы** (Лёгкий / Средний / Тяжёлый), чтобы оценка времени и дневной план учитывали, что темы разной трудоёмкости. Сложность выбирает студент сам (редактируемое поле темы); значение по умолчанию — «Средний».

## 2. Область (scope)

**Входит (этот инкремент, только `ExamPlanner.Core`):**
- Модель: enum `TopicDifficulty` + поле `Topic.Difficulty`.
- Логика: расчёт эффективных минут на тему по сложности в `PlannerService`.
- Юнит-тесты на новую логику и на обратную совместимость.

**Не входит (отдельные задачи):**
- UI выбора сложности на экране темы (`EditTopic`/`ExamDetail`) — делает frontend-агент позже.
- Отображение сложности в списках/плане — frontend.
- Конфигурируемые коэффициенты, доступность по дням недели, таймер — отдельные фазы роадмапа.

## 3. Модель данных

Новый enum (порядок значений выбран под бесшовную миграцию):

```csharp
public enum TopicDifficulty
{
    Medium = 0,
    Easy = 1,
    Hard = 2
}
```

`Medium = 0` — потому что sqlite-net при добавлении новой колонки в существующую таблицу присваивает старым строкам `0`. Значит все ранее созданные темы автоматически становятся «Средний» без отдельной data-миграции.

В `Topic` добавляется поле:

```csharp
public TopicDifficulty Difficulty { get; set; } = TopicDifficulty.Medium;
```

Остальные поля `Topic` без изменений. `Exam.MinutesPerTopic` сохраняется и трактуется как **базовая оценка для «средней» темы**.

## 4. Расчёт эффективного времени

Коэффициенты (фиксированные константы в `PlannerService`, не настраиваемые — YAGNI):

| Сложность | Коэффициент |
|-----------|-------------|
| Лёгкий (Easy)   | 0.5 |
| Средний (Medium)| 1.0 |
| Тяжёлый (Hard)  | 1.5 |

Эффективные минуты на тему (целое, округление к ближайшему, `.5` — от нуля):

```
EffectiveMinutes(topic, exam) =
    (int)Math.Round(exam.MinutesPerTopic × Multiplier(topic.Difficulty),
                    MidpointRounding.AwayFromZero)
```

`MidpointRounding.AwayFromZero` — явно, чтобы избежать банковского округления по умолчанию (например `25 × 0.5 = 12.5 → 13`, а не `12`). Пример при `MinutesPerTopic = 30`: Лёгкий → 15, Средний → 30, Тяжёлый → 45.

Возвращается `int`, чтобы `PlanItem.Minutes` (int) и бюджет дневного плана оставались согласованными без изменения публичных типов.

## 5. Изменения в PlannerService

- **`BuildPace`** — `NeedMinutesPerDay` теперь = `Σ EffectiveMinutes(t, exam)` по **оставшимся** темам `÷ daysLeft` (раньше: `осталось × MinutesPerTopic ÷ daysLeft`).
- **`BuildDailyPlan`** — при наборе дневной квоты бюджет времени и `PlanItem.Minutes` считаются по `EffectiveMinutes(topic, exam)` каждой темы (раньше — плоское `exam.MinutesPerTopic`). Порядок набора (по срочности экзамена, затем `Position`) и флаг «не помещается» — без изменений.
- **`TodayQuota`** — **без изменений**: квота остаётся по количеству тем (`ceil(осталось / дней)`). Сложность влияет только на минуты, не на число тем в день. (Договорённость владельца; при необходимости пересмотрим позже.)
- **`BuildDashboard`** — логика без изменений: суммирует `p.NeedMinutesPerDay`, который теперь уже взвешен по сложности.

Публичные сигнатуры методов и record-типы (`ExamPace`, `PlanItem`, `DailyPlan`, `DashboardSummary`) **не меняются** — меняется только вычисление минут внутри.

## 6. Обратная совместимость и миграция

- Новая колонка добавляется автоматически при `CreateTableAsync<Topic>()` (self-init репозитория). Старым строкам достаётся `0` = `Medium`.
- При всех темах «Средний» (×1.0) все расчёты дают **те же числа, что и до фичи** → существующие 14 тестов остаются зелёными без правок.

## 7. Тестирование

Новые юнит-тесты (`ExamPlanner.Core.Tests`), детерминированные:
- `EffectiveMinutes`: Easy/Medium/Hard дают round(base×0.5/1.0/1.5) для нескольких баз (30, 25, 40).
- `BuildPace`: смешанные сложности → `NeedMinutesPerDay` = сумма эффективных минут оставшихся тем ÷ дни; тема «Done» не учитывается.
- `BuildDailyPlan`: тяжёлые темы съедают больше бюджета; `PlanItem.Minutes` соответствует сложности; проверка флага «не помещается», когда тяжёлые темы не влезают.
- Обратная совместимость: экзамен со всеми «Средний» даёт числа, идентичные текущему поведению.
- Дефолт: новая `Topic` без явной сложности = `Medium`.

## 8. Допущения

- Коэффициенты 0.5 / 1.0 / 1.5 фиксированы в коде; изменение — правка константы.
- Округление эффективных минут — к ближайшему целому.
- Квота по количеству (не по времени) — осознанное упрощение на этот инкремент.
- UI выбора/отображения сложности появится отдельно (frontend); до этого все новые темы — «Средний».
